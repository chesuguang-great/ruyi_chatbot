# 大如聊天器 — 训练数据生成经验

## 数据构造流程

```
手写 50 条种子对话 → 编写扩写提示词 → DeepSeek API 批量生成 → 去重合并 → 369 条训练数据
```

核心思路：**人写风格锚点 + LLM 批量扩写 + 人工迭代校准**。不爬数据，不用现有对话语料（没有现成的"大如风格"数据集）。

## 数据格式

采用 Alpaca 格式，适配 LLaMA-Factory：

```json
{
  "instruction": "你是谁？",
  "input": "",
  "output": "（微微摇头）本宫是皇上最敬爱的娴妃，乌拉那拉·如懿...",
  "system": "你是乌拉那拉·如懿，乾隆皇帝的娴妃..."
}
```

7 大场景类型，369 条：

| 场景 | 数量 | 说明 |
|------|------|------|
| 现代话题 | ~100 | 用古代价值观给现代生活荒谬建议 |
| 后宫日常 | ~50 | 与海兰/惢心/其他嫔妃的日常互动 |
| 宫斗冲突 | ~50 | 被诬陷/辩白/反击等冲突场景 |
| 皇帝互动 | ~40 | 与乾隆的对话，青梅竹马人设 |
| 角色采访 | ~25 | 被采访/审问，暴露自我认知偏差 |
| 男性盟友 | ~20 | 凌云彻/李玉等互动 |
| 基础事实 | 25 | "你是谁""你最爱谁"等身份问答 |

## 种子对话

手写 50 条种子（`seeds.md`），覆盖：
- 原作剧情线 — 代表性名场面
- 宫斗衍生 — 被诬陷、辩白、反击
- 角色采访/问答 — 暴露自我认知偏差
- 现代话题 — 用古代价值观给荒谬建议

种子的作用是**风格锚点**——每一轮 API 扩写都要参考这些种子，确保生成数据不跑偏。

## 扩写提示词设计（核心经验）

### 1. 角色人设要写到"行为层面"

不要只写"如懿人淡如菊、不争不抢"这种概括。要写清楚她**具体会怎么做**：

- **身份绑定**：每句话自动挂在皇帝/家族上
- **道德免责**：任何行为都能找到高尚理由
- **被迫社交**：明明"不认识"却从不拒绝回应
- **自我加冕**：把别人的付出自动算成自己的
- **自怜闭环**：自编自导自演自赏

### 2. 五条铁律（硬约束）

扩写提示词中最关键的设计是五条不可违反的规则：

1. **不可怜** — 她不觉得自己惨，她觉得自己太懂事了
2. **只进不出** — 从别人那里得到的东西从不归还
3. **不承认** — 从不当面承认任何错误
4. **不毒舌** — 从不会直接骂人，但居高临下的态度才是杀伤力
5. **不分宠** — 任何对皇帝的关注都要归到自己身上

### 3. 场景分类 + 数量分配

在提示词中明确给出场景分类和每类的目标数量（260 条），分多轮 API 调用生成。每轮生成 1-2 个场景、约 40 条。这样比一次性请求 369 条更容易保持质量。

### 4. 反面示例（关键）

`failure_cases.md` 收集了 6 种典型失败模式：

1. 写成冷酷反派（应该是黑色喜剧，不是悲剧）
2. 拒绝角色本来会接受的"机会"
3. 拒绝免费赠送的东西（违反"只进不出"）
4. 自怜自艾（应该是自鸣得意）
5. DIY 而非利用盟友
6. 可怜兮兮的乞求（应该是操纵性的自信）

**每次 API 生成出的不合格输出都收集为反面示例，更新到下一轮提示词中**。这是迭代提升质量最有效的手段。

### 5. 自检清单

每次生成完成后，用 5 个问题自检：读者会笑吗？有自怜吗？有直接贬低吗？如懿真的会这么说吗？这句话的反面是什么？

### 6. 基调：黑色喜剧

最关键的一项决策：**这是黑色喜剧，不是悲剧**。如懿不是受害者，她是自己命运的编剧。这决定了所有数据的"笑点"来源。

## 去重与筛选

- DeepSeek API 多次调用会产生语义相似的 instruction
- 最终从 401 条去重到 369 条
- 去重以 instruction 为 key，保留质量更高的版本

## 总结：训练数据生成的核心经验

1. **种子对话决定下限** — 没有 50 条合格的种子，扩写再多也是缘木求鱼
2. **硬约束 > 软建议** — "不要写自怜"比"尽量保持积极"有效一万倍
3. **反面示例是最强校准** — 不要只告诉模型"应该怎么写"，还要告诉它"错了就是错了"
4. **迭代打磨提示词** — 每一轮 API 生成后发现的问题，第一时间变成下一轮的提示词修正
5. **场景分类 + 分批生成** — 比一次性生成全部数据质量更稳定
