# 大如聊天器 — 微调技术总结

## 项目目标

使用 LoRA 微调 Qwen2.5-7B-Instruct，训练一个模仿《如懿传》"大如"角色的娱乐聊天机器人。根本目的是通过这个有趣的项目学习 LoRA 微调的完整流程。

## 技术选型与原因

| 层 | 选型 | 原因 |
|---|------|------|
| 基座模型 | Qwen2.5-7B-Instruct | 中文 SOTA，7B 大小可装入 24GB 显存 |
| 微调框架 | LLaMA-Factory | 一站式 SFT + LoRA，封装完善，上手快 |
| 微调方式 | LoRA（rank=32, alpha=64） | 只训 ~1% 参数，降低显存和过拟合风险 |
| 推理引擎（本地） | Ollama | 一键部署，Metal 加速（M5 GPU），统一本地/云端方案 |
| 推理引擎（训练平台） | PyTorch + PEFT | 训练和合并权重 |
| 聊天界面 | Gradio | 最快出网页聊天 UI，支持 share 公网链接 |
| 模型量化 | GGUF Q4_K_M | 15GB → 4.4GB，精度损失可接受 |

## 为什么必须微调而不是 Prompt/RAG

大如是带有强烈负面/偏执性格的角色。如果只靠 Prompt：
- Qwen2.5 的安全对齐会逐渐把角色"拽回"治愈系
- 聊多了会开始"劝你向善""理解他人"
- 角色稳定性无法保证

**LoRA 微调直接修改模型权重**，让角色行为变成"本能"而非"指令"，才能稳定维持人设。

## LoRA 原理简述

LoRA（Low-Rank Adaptation）的核心思想：

- 冻结原模型全部权重（7B 参数不动）
- 在每个目标层旁插入一对低秩矩阵 A 和 B（rank=32）
- 前向传播时：`output = Wx + BAx`
- 只训练 A 和 B 矩阵（~1% 参数量）
- 显存和计算量大幅降低，但对特定任务的拟合效果接近全参微调

公式：`ΔW = BA`，其中 `B ∈ R^(d×r)`，`A ∈ R^(r×k)`，`r << min(d, k)`

## 训练配置

### LoRA 参数

| 参数 | 值 | 说明 |
|------|-----|------|
| lora_rank | 32 | 低秩矩阵的秩，越大拟合能力越强但参数越多 |
| lora_alpha | 64 | 缩放因子，alpha/rank 控制 LoRA 权重的影响力 |
| lora_dropout | 0.05 | 防止过拟合 |
| lora_target | all | Qwen2.5 所有线性层（q/v/k/o/gate/up/down proj） |

### 训练超参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 阶段类型 | SFT | 监督微调 |
| 精度 | bf16 | Brain Float 16，保持精度同时降低显存 |
| 学习率 | 5e-5 | LoRA 通常比全参微调学习率稍高 |
| 学习率调度 | cosine + warmup 10% | 前期线性升温，后期余弦衰减 |
| epoch | 3 | 数据量 369 条，适当多训几轮 |
| 实际完成 epoch | 2.98 | 接近 3 轮 |
| batch size（有效） | 8 | per_device=2 × gradient_accumulation=4 |
| cutoff_len | 1024 | 最大序列长度 |
| max_grad_norm | 1.0 | 梯度裁剪 |
| 优化器 | Adam (β1=0.9, β2=0.999) | 标准选择 |
| 随机种子 | 42 | 可复现 |

## 训练执行

- **GPU**：RTX 5090（实际使用，原计划 RTX 3090/4090）
- **平台**：AutoDL（国内 GPU 云平台）
- **训练耗时**：约 111 秒（1.85 分钟）
- **吞吐**：9.98 samples/s，1.24 steps/s
- **最终 loss**：1.648
- **LoRA 权重大小**：308 MB（adapter_model.safetensors）


## 模型部署流水线

本地 Mac（M5 + 24GB 内存）部署流程：

```
LoRA 权重（308MB）
  + Qwen2.5-7B-Instruct（14GB）
    → merge_lora.py（merge_and_unload）
      → 完整 PyTorch 模型（15.2GB, float16）
        → llama.cpp convert_hf_to_gguf.py
          → F16 GGUF（15.2GB）
            → llama-quantize Q4_K_M
              → Q4_K_M GGUF（4.4GB）
                → ollama create ruyi
                  → ollama run ruyi / API 调用
```

### 各步说明

| 步骤 | 工具 | 输入 | 输出 | 耗时 |
|------|------|------|------|------|
| 合并权重 | merge_lora.py (PyTorch + PEFT) | LoRA adapter + base model | 15.2GB safetensors | ~1.5 分钟 |
| 转 GGUF F16 | llama.cpp convert_hf_to_gguf.py | safetensors | 15.2GB GGUF | ~17 秒 |
| Q4_K_M 量化 | llama-quantize | F16 GGUF | 4.4GB Q4_K_M | ~36 秒 |
| 创建 Ollama 模型 | ollama create | Q4_K_M GGUF + Modelfile | Ollama 模型 | ~20 秒 |

### Q4_K_M 量化说明

- Q4_K_M 是 llama.cpp 推荐的"性能和大小最佳平衡"量化方案
- 4-bit 量化，但关键层（attention、output）使用更高精度
- 15.2GB → 4.4GB（约 3.3x 压缩）
- BPW：16.0 → 4.91
- 实际推理精度损失感知不到

## Gradio 聊天界面

- 调用 Ollama `/api/chat` 接口
- 适配了新版 Gradio 6.x 的 ChatInterface（history 格式变化）
- `--share` 参数直接生成公网链接
- 本地推理速度：M5 + 4.4GB Q4_K_M，约 2-5 秒/轮

## 踩坑记录

1. **为什么必须微调**：Prompt/RAG 被 AI 的安全对齐拽回"治愈系"。LoRA 直接改权重才稳定。
2. **不自怜是关键坑**：API 容易生成"心里苦""叹气""凄楚"等自怜动作，需在提示词中反复强调并给反面示例。
3. **数据格式适配**：DeepSeek API 偶发 JSON 格式不规范（被 markdown 代码块包裹），脚本做清理处理。
4. **去重必要**：多轮 API 调用的 instruction 有语义重复，从 401 条去重到 369 条。
5. **Python SSL 兼容**：macOS 系统 Python 3.9 LibreSSL 无法连接 HuggingFace，需要 pyenv + venv 管理 Python 环境。
6. **Gradio 6.x 兼容**：新版 Gradio 的 ChatInterface 回调签名和 history 格式都有变化，content 从字符串变成了 `[{"text": "...", "type": "text"}]` 嵌套结构。
