## 主要问题

### 1. `app.py` 与当前 Gradio 接口约定不一致

- 文件：`app.py`
- 严重级别：高
- 问题：
  - `respond()` 返回 `("", history)`，这是旧式聊天回调写法。
  - `format_chat()` 假设 `history` 一定是 `list[list[str]]`。
  - 但项目依赖声明是 `gradio>=4.0.0`，当前版本的 `ChatInterface` 常见 history 格式是字典列表，回调一般返回单条 assistant 文本。
- 影响：
  - 本地 PyTorch 推理入口很可能直接因 history 解包失败或返回值格式不匹配而不可用。
- 建议：
  - 参考 `app_ollama.py` 的 history 兼容逻辑，统一改成兼容新版 Gradio 的实现。
  - 把三套前端入口的消息格式处理抽成共享函数，避免重复分叉。

### 2. `start_train.sh` 训练数据路径写错

- 文件：`start_train.sh`
- 严重级别：高
- 问题：
  - 脚本从项目根目录复制 `dataset_ruyi.json` 和 `dataset_info.json`：
    - `cp "$SCRIPT_DIR/dataset_ruyi.json" ...`
    - `cp "$SCRIPT_DIR/dataset_info.json" ...`
  - 但仓库中的实际路径在 `train_datas/` 下：
    - `train_datas/dataset_ruyi.json`
    - `train_datas/dataset_info.json`
- 影响：
  - 按当前文档直接执行训练脚本时，会在复制数据阶段失败。
- 建议：
  - 改为从 `train_datas/` 读取。
  - 同步修正文档中“上传 4 个文件到根目录”的描述，避免脚本和文档继续分裂。

### 3. `requirements.txt` 没覆盖所有已提供入口

- 文件：`requirements.txt`
- 严重级别：中
- 问题：
  - 当前只包含：
    - `torch`
    - `transformers`
    - `peft`
    - `gradio`
    - `accelerate`
  - 但仓库里其他脚本还依赖：
    - `httpx`：`app_ollama.py`
    - `openai`、`python-dotenv`：`generate_training_data.py`
    - `numpy`、`safetensors`：`convert_lora_to_mlx.py`
    - `mlx-lm`：`app_mlx.py`
- 影响：
  - 用户按当前依赖安装后，运行多个官方入口会直接报 `ModuleNotFoundError`。
- 建议：
  - 至少补齐通用依赖。
  - 如果 MLX 仅用于 Apple Silicon，可拆成额外依赖说明，如 `requirements-mlx.txt`。

### 4. `start_train.sh` 的兜底训练分支无效

- 文件：`start_train.sh`
- 严重级别：中
- 问题：
  - 兜底分支调用：
    - `run_exp({'train_config.yaml'})`
  - 这里传入的是 Python `set`，不是合理的配置参数。
- 影响：
  - 一旦前两个启动分支不可用，最终兜底逻辑本身就会失败。
- 建议：
  - 删除这段无效分支，保留一个明确、已验证的调用方式。
  - 或按当前 LLaMA-Factory 版本改成真实可运行的 API/CLI 调用。

### 5. `start_train.sh` 的数据集注册逻辑实际上没有执行合并

- 文件：`start_train.sh`
- 严重级别：中
- 问题：
  - 注释写的是“合并 dataset_info（不覆盖已有的）”。
  - 实际行为是先覆盖复制 `dataset_info.json`，后面的 Python 代码只读取并打印，没有写回任何合并结果。
- 影响：
  - 如果目标环境已有其他数据集注册项，当前流程可能会把它们覆盖掉。
  - 脚本日志与真实行为不一致，后续排障成本高。
- 建议：
  - 明确选择一种策略：
    - 真正合并并写回
    - 或直接覆盖，但注释和日志必须说清楚

### 6. `app_mlx.py` 的 history 内容提取不稳

- 文件：`app_mlx.py`
- 严重级别：低
- 问题：
  - `format_prompt()` 直接把 `item.get("content", "")` 插进 prompt。
  - 新版 Gradio 的 `content` 可能是嵌套结构，不一定是纯字符串。
- 影响：
  - prompt 中可能混入 Python 对象字符串，导致模型输入污染。
  - 这和文档里提到的“MLX 本地推理格式提取不稳定”现象相符。
- 建议：
  - 复用 `app_ollama.py` 里类似 `_extract_text()` 的逻辑。
  - 对三套推理入口统一做 message normalize。

## 次要观察

### 1. 入口实现重复较多

- `app.py`、`app_ollama.py`、`app_mlx.py` 都维护了一份几乎相同的 system prompt。
- 后续修改角色设定时，很容易出现三处不一致。
- 建议把 prompt 和消息格式化逻辑提取到共享模块。

### 2. 文档与现状有分叉

- `PLAN.md` 中仍保留部分阶段性说明，例如“明天弃用 MLX，改用 Ollama”。
- 但仓库实际同时保留了 PyTorch / MLX / Ollama 三套入口。
- 建议后续补一份“当前推荐运行方式”文档，避免使用者误判。

## 建议修复顺序

1. 修 `app.py` 的 Gradio 接口兼容问题。
2. 修 `start_train.sh` 的数据路径和无效兜底分支。
3. 补全 `requirements.txt`，确保所有公开入口可安装。
4. 统一三套推理入口的 message/history 处理。
5. 清理文档分叉，明确推荐推理方案。
